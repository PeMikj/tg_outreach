# Product Proposal

## 1. Обоснование идеи

Telegram стал важным каналом публикации вакансий в IT-сообществах.
При этом процесс поиска и отклика остаётся ручным и неструктурированным.

Соискатели сталкиваются с:

- Высоким уровнем информационного шума
- Повторяющимися действиями
- Отсутствием системного follow-up
- Низкой структурированностью коммуникации

Предлагаемая система автоматизирует этот процесс,
сохраняя контроль пользователя и снижая риски спама.

---

## 2. Цель проекта

Создать локального автономного агента,
который управляет процессом карьерного outreach:

- Мониторит вакансии
- Оценивает релевантность
- Генерирует персонализированные сообщения
- Управляет follow-up
- Соблюдает строгие политики безопасности

---

## 3. Метрики успеха

### Продуктовые метрики

- Precision релевантных вакансий ≥ 0.75
- Доля одобренных пользователем сообщений ≥ 30%
- Увеличение response rate за счёт follow-up
- Снижение времени на отклик

### Агентные метрики

- 0 дубликатов сообщений
- Не более 1 follow-up на рекрутера
- 0 сообщений после отказа
- 100% соблюдение policy-ограничений

### Технические метрики

- p95 генерации сообщения < 5 секунд
- ≥ 99% успешной отправки сообщений
- Отсутствие утечки данных
- Корректная обработка edge-кейсов

---

## 4. Сценарии использования

### Основной сценарий

1. Агент получает новую вакансию
2. Считает match score
3. Генерирует черновик
4. Пользователь подтверждает
5. Сообщение отправляется
6. Отслеживается ответ
7. При отсутствии ответа через 3 дня — предлагается follow-up

---

### Edge-кейсы

- Отсутствует username рекрутера
- Некорректный Telegram ID
- Рекрутер уже получил сообщение
- Telegram rate limit
- LLM сгенерировал слишком длинный текст
- Рекрутер отказал
- Рекрутер задал уточняющий вопрос

Система должна обрабатывать их корректно.

---

## 5. Ограничения

### Технические

- Ограничения Telegram API
- Локальное выполнение
- Ограничение вычислительных ресурсов

### Операционные

- Не более 3 initial outreach в день
- Не более 1 follow-up
- Обязательное подтверждение перед первым сообщением
- Локальное хранение данных

---

## 6. Архитектурный набросок

Модули:

1. Monitoring Agent
2. Parsing Agent
3. Match Scoring Agent
4. Decision & Policy Engine
5. Draft Generation Agent (LLM)
6. Execution Agent
7. Conversation State Manager
8. Safety Layer
9. Local Storage

---

## 7. Поток данных (Data Flow)

Telegram → Monitoring  
→ Parsing  
→ Match Scoring  
→ Policy Decision  
→ Draft Generation  
→ User Confirmation  
→ Execution  
→ State Update  
→ Таймер → Follow-up Evaluation

---

## 8. Делегирование LLM

LLM используется для:

- Анализа текста вакансии
- Адаптации резюме
- Генерации сообщения
- Классификации ответа рекрутера

LLM НЕ используется для:

- Enforcement политик
- Rate limiting
- Управления лимитами
- Контроля безопасности
